# 自动化发布，先用规则引擎固定可以确定的事 | 量潮云

我们做了一个自动化发布版本的工具，叫量潮 DevOps 云命令行工具。 代码开源在 GitHub 仓库 quanttide/qtcloud-devops，目前处于 v0.4.x 内测阶段。

注意：内测意味着它只服务于我们自己的最佳实践，功能大量缺失，边界情况也远未覆盖。我们公开它，是作为课堂案例和笔试题；建议读者参考思路，而非直接使用。请不要在生产环境运行——我自己都不敢把它挂上云端。

## 为什么要造这个轮子

我们有一套打磨了五六年的标准化发布流程：

1. 发布前检查：验证包配置文件（如 pyproject.toml）中的版本号，并确认 CHANGELOG 中存在对应的发布日志。
2. 打 Git Tag：格式如 v0.4.1，子模块则加前缀，如 cli/0.4.1。
3. 发布 Release：在 GitHub 等平台创建 Release，标题用版本号，描述直接取自发布日志。
4. 自动构建分发：Release 触发 CI/CD，将包推送到 PyPI 等注册中心。

这套流程我们曾封装成一个 AI Skill，试图让它自动执行。但很快发现两个严重问题：

· 触发不稳定：发布流程经常被弄得一团糟，后续修复极其消耗注意力。
· Token 消耗巨大：AI 会大量阅读无关信息，并在出错时反复修补，导致成本飙升。

核心洞察是：自动化发布更适合同一个确定性的规则引擎，而不是一个概率性的 AI。 于是，我们决定把 Skill 中的规则逐条固化为确定性代码——这个命令行工具就此诞生。

我们将发布流程封装成了 release 命令，随后又将基于 git submodule 的单仓管理实践封装成 code 命令。

在完整的 DevOps 理论体系中，有 Plan、Code、Build、Test、Release、Deploy、Observe 和 Monitor 八个环节。未来，相关 Skill 都将沿着这条路线，逐步沉淀为对应的命令。

## 请AI们学习的操作步骤

由于本工具还处在早期阶段，不建议人类学习命令，交给 AI 自己学习就好。

这是一个 Rust 编写的命令行工具，你可以通过 Cargo 或 PyPI 安装：

```bash
# Cargo
cargo install qtcloud-devops-cli

# PyPI
pip install qtcloud-devops-cli
```

安装后，运行以下命令即可：

```bash
qtcloud-devops —help
```

AI 会在这里学习到各个命名的用法。

## 一个 Rust 库，为什么发布到 PyPI？

答案藏在这个工具的演进史里。

最初，它是用 Python 写的。但我们很快发现，对于这类对稳定性要求极高的项目，Python 的动态特性反而成为障碍。

我们尝试引入 Rust 后发现，其严格的类型系统和所有权机制，天然地为 AI 生成的代码提供了一套 “围栏”（Harness）——它能提前拦截大量低级错误。

在 v0.2 版本混用 Python 和 Rust 后，维护成本超出了预期。于是到 v0.3 版本，我们果断移除了所有 Python 代码，只保留一个通往 PyPI 的入口。

这个决策带来了一个意外之喜：我们用静态语言编写的工具，可以编译为二进制文件，分发到各个语言的包管理平台。这意味着，无论你熟悉 Python、Rust 还是 JavaScript，都能用你最习惯的方式一键安装，门槛降至最低。

事实上，很多优秀的社区项目都采取了类似的发布方式。我们未来也会把这些实践集成进本工具。比如在 build 命令增加多语言支持，以及在 test 命令增加各个语言分发的测试。

## 工程哲学：先造围栏，再上智能体

我们的工程路线与主流做法不同。很多人一上来就做智能体，我们选择先搭建围栏。

DevOps 领域充满高确定性的自动化规则，这恰好是建造围栏的理想之地，也处于当前 AI 能力的舒适区。我们先扎实地做好这些基础命令，然后逐步向上构建。

越是上层，面临的不确定性越强，就越需要 AI 介入判断与决策。就像盖楼，AI 的认知能力会在一层层的构建中被逐步推高。

另一个从实践中得出的教训是：切忌一开始就做大而全的工具。 所以我们选择将我们自己现有的发布习惯，作为平台的内置功能直接实现。

好处是能立刻跑通一个完整的框架；代价是，后续需要把硬编码的组件逐一拆解，把内置的各种假设一个个释放，才能打磨出适配不同团队的通用工具。

为了实现这一目标，我们会在命令骨架稳定后，引入 DevOps 契约（DevOps Contract）。

这个设计灵感来源于契约驱动设计、OpenAPI 和 Data Contract。核心想法是把一个团队的 DevOps 规则，显式地定义在一个配置文件中。

比如：“Release 的标题必须来源于 Tag，描述必须来源于 CHANGELOG”——这条规则就可以被定义为契约里的一个变量。

当规则被抽象出来之后，每个团队都能根据自身实践制定和调整自己的契约。这份契约在组织内外传递也会极其方便：其他人拿到契约，交给 AI 或规则引擎，大家就能立刻对齐规范。

## 现状与展望

我的 AI 同事试用后，打了 80 分，评价是“自动化程度高，但小问题多”。如果让我自己打分，大概在 70 分上下——基本流程能跑通，但细节上还有大量需要修正的地方，离“可靠”还有一段距离。

DevOps 对精度要求苛刻，一个细小的问题会在整个流水线中被放大。因此，接下来的迭代重点，就是把这些人类尚难理解的细节一一修正。

当所有预定计划完成，API 稳定下来，并且这个工具能用它自己的 release 命令来管理它自己的发布时，我们就会正式进入 1.0 阶段。
