# 量潮DevOps云命令行发布v0.9.2

这篇文章探讨最近一段时间的进展。

## 当前状态和后续计划


目前 DevOps 云命令行工具（quanttide/qtcloud-devops）已经覆盖了一个命令行工具所 DevOps 生命周期的前五个阶段，包括 plan、code、build、test、release。后三个阶段拟在有服务端需求的时候开始制作。


根据我们的 DevOps 理论体系制作的 DevOps 契约命令 contract，把隐式的规范扫描成显性的规范，或者也可以自定义契约作为规范。


由于本命令行工具依赖了大量的外部工具，包括但不限于 git、gh、uv、cargo 等十几个工具，未来可能还会更多，因此还准备了一个 doctor 命令专门维护外部依赖。


v0.10 及后续几个版本拟更新 release 阶段自动判断版本号，并以此为切入点改进自动生成和维护 commit的 code 阶段逻辑。


## 演化和迭代历程


这个命令行工具是从发布开始的。


一开始的起点是要自动化一系列枯燥且容易出错的版本发布任务：更新版本号、更新 CHANGELOG、更新 Git Tag、更新 GitHub release。


后来在封装自己的命令 Skill 的过程中发现了很多负面影响，于是不得不继续回到前面的流程不断增加环节。从 0.4-0.9，plan、code、build、test 也逐渐增加进来。


plan 命令是原创的逻辑，使用 ROADMAP 作为事实源，来源于我的开发习惯。未来也会增加 TODO 用以分解版本目标为具体任务。


code 命令是对 git 提交等代码同步逻辑的封装。 build、test 命令则主要是封装各语言命令行的 build 工具和 test 工具。


在这个过程中，预设的 DevOps 契约开始浮现，AI 自己也独立地设计出来了一些逻辑，于是我们系统地引入了 contract 显性化相关概念。


到目前为止，整个平台已经具备了较强的自举能力，patch 版本级别的更新几乎不需要人工干预，minor 版本级别的更新也只需要合适的上下文作为导航。


## 为什么要造研发流程自动化工具？


AI 产出能力太强，因此必须要通过研发流程自动化工具约束和 AI 的交付行为。


DevOps 理论体系提供了一整套完整的框架，方便我们在此基础上进行整合。


在这套框架下，我们可以把一次迭代约束为一轮版本更新，这样就可以把复杂项目的研发决策分解为一个一个 AI 擅长理解的、人类也可以评审和把握的阶段。


虽然 AI 和人类不能解决所有的问题，但是在研发流程自动化过程中，我们可以发现很多观测系统健康度的指标，比如代码圈复杂度、静态检测、测试覆盖率等等。


有了这些异常指标，系统就具备了一个纠错的目标。在探究代码质量为什么糟糕的过程中，真实的代码异常问题往往容易被发现，特别是不符合业务预期的原因。


有了严格的研发流程自动化工具链为基础，海量地智能体在后台并行写不同项目也成了可能。这套工具可以像一个高效的工厂批量发布 AI 产出结果，快速进入业务环境验证。


在这个基础上，整个量潮云的其他子系统也可以加快迭代，并且很快反馈到 DevOps 云的迭代，形成良好的正反馈自举结构。
